# 커뮤니티 공간 — 전체 유저 장소/코스 공유

> 연관: `05-public-enrichment.md`(F 항목·Phase 3와 통합 — 아래 "05와의 관계" 참조), `docs/DECISIONS.md`(2026-07-07 익명화 결정과의 텐션).
> 목적: 지금은 **커플 스코프 기록의 공개 토글**로만 `/explore`에 노출된다. 사용자는 "전체 유저들이 장소를 공유하는 **커뮤니티 공간**"을 원한다. 즉 커플 단위를 넘어 **모든 유저가 장소/코스를 발견·공유·상호작용**하는 커뮤니티 성격을 얹는다.

## 배경 / 왜 지금
- 공개 표면(0006–0008)은 이미 **anon-safe 뷰 + copy-on-publish** 패턴으로 라이브다: `/explore`(코스 피드), `/explore/[id]`(코스 상세), `/places`·`/places/[id]`(장소 디렉터리/상세). 크롤 가능한 공개 URL 축은 섰다.
- 그러나 지금 공개 표면은 **일방향 전시(display-only)**다. 방문자가 발견은 하지만 **참여(좋아요·저장·따라하기·작성자 회유)** 수단이 없다 → 재방문·체류·커뮤니티감이 없다.
- 05 문서의 **F(서비스 자체 풍부화)**·**Phase 3**가 정확히 이 영역인데 "Phase 3, 사용자 결정 대기"로 유보돼 있다. 이 문서가 그 F/Phase3를 **"커뮤니티"라는 하나의 방향으로 재정의**하고 로드맵을 구체화한다.

## 현황 진단 (2026-07-09)

### 지금 가능한 것
- 커플이 로그를 작성 → `visibility=public` 토글 → `public-covers`에 커버 복사 → anon-safe 뷰(`explore_logs`/`explore_log_places`/`explore_places`/`explore_place_logs`)로 전 세계 열람.
- **작성자는 완전 익명**(0007/DECISIONS 2026-07-07): 공개 표면 어디에도 닉네임/신원 없음. "익명 커플"조차 표시 안 함.
- 프라이버시 원칙: memo·갤러리·private cover·route 폴리라인은 **구조적으로** 비노출. 원본 무손상, 되돌림 가능.

### 커뮤니티로서 비어있는 것
| 축 | 현재 | 커뮤니티에 필요 |
| --- | --- | --- |
| 참여(engagement) | 없음(전시만) | 좋아요/북마크/따라하기/(댓글?) |
| 발견 동선 | 최신 desc 나열 | 인기·큐레이션·팔로우 피드 |
| 정체성(identity) | 완전 익명 | **작성자 표시 여부가 핵심 텐션 — 아래 열린질문 1** |
| 소속감 | 커플 사적 공간 | 커뮤니티 피드/보드/프로필 |
| 상호작용 신호 저장 | 없음 | 좋아요/조회 카운트 테이블(+RLS) |

### 근본 텐션 (반드시 사용자 결정 선행)
커뮤니티는 본질적으로 **"누가 올린 콘텐츠"**라는 사회적 신호로 굴러간다(팔로우·인기 크리에이터·신뢰). 그런데 우리는 **작성자 완전 익명(0007)**을 이미 결정·구현·라이브했다. 이 둘은 정면 충돌한다. 커뮤니티를 제대로 하려면 익명화 결정을 **부분적으로라도 뒤집는 ADR**이 필요하다. 강행 불가 — 열린질문 1에서 세 가지 모델을 제시한다.

## 방안 후보 (레이어별)

각 레이어는 **프라이버시 원칙**(memo/갤러리/route/정확위치 비노출, copy-on-publish, 공개 로그에만 상호작용, RLS-first)을 계승한다.

### L1. 참여 신호 — 좋아요 + 북마크 (커뮤니티의 최소 단위, 최우선)
- **좋아요**: 공개 코스/장소에 하트. 05-D의 "인기 신호" 겸용. 로그인 필요, **카운트는 공개**(anon 뷰에 집계만).
- **북마크(저장)**: 마음에 든 공개 코스를 내 보관함에. 개인 소유 데이터(로그인 사용자 본인만).
- **데이터모델(신규 마이그레이션 `0009_community_signals`, public/test 미러 + SCHEMA.md):**
  - `date_log_likes(date_log_id, user_id, created_at)` — PK(date_log_id,user_id). RLS: 본인 행 insert/delete, 대상은 `visibility=public` 로그만. select는 본인 것만(내가 눌렀는지). **집계는 뷰로.**
  - `date_log_bookmarks(date_log_id, user_id, created_at)` — 동형. select 본인 것만.
  - anon-safe 집계는 기존 `explore_logs`에 `like_count` 파생 컬럼 추가(뷰 재정의) 또는 `explore_log_stats(date_log_id, like_count)` 별도 뷰. **개별 좋아요 누른 사람 신원은 anon에 절대 노출 안 함**(카운트만).
- **프라이버시:** 좋아요 대상은 공개 로그 한정. 누가 눌렀는지는 비공개(카운트만 공개). 북마크는 순수 개인 데이터.
- **난이도:** 중. 의존: 없음(0008 위에 얹음). **커뮤니티의 진입 레이어.**

### L2. "이 코스 따라하기" (copy-on-publish의 역방향 — 전환·재방문 최강)
- 공개 코스를 **내 초안 date-log로 복제**(장소 목록·순서만 복사, memo/사진/평점은 복사 안 함 — 내가 새로 채움).
- **데이터모델:** 신규 테이블 불필요할 수 있음. `date_logs`에 `forked_from uuid null`(출처 표기·집계용) 추가 정도. 복제는 서버 액션/RPC로 공개 뷰→내 커플 초안 생성.
- **사용자가치:** "이 데이트 코스 우리도 가보자" → 즉시 내 기록 시작 → 강력한 재방문·전환 루프.
- **AdSense/SEO:** 직접 URL 증식은 아니나 체류·전환 급상승. `forked_from`로 "N명이 따라한 코스" 인기 신호 확보.
- **난이도:** 중. 의존: L1 이후 권장(따라하기 수도 인기 신호). **프라이버시:** 복제는 안전 컬럼(장소/순서)만.

### L3. 커뮤니티 피드/보드 + 큐레이션 정렬
- `/community`(또는 `/explore` 확장): **최신 / 인기(좋아요순) / 지역 / 카테고리** 탭. 05-C(지역), 05-D(정렬)를 커뮤니티 홈으로 통합.
- 큐레이션: 에디터 픽(수동 플래그 `date_logs.featured bool`) 또는 좋아요 상위 자동.
- **난이도:** 저(최신·인기 정렬은 L1 카운트 위에서 즉시) ~ 중(전용 `/community` IA). 의존: L1.
- **헤더 진입점**은 `07-header-footer.md`에서 배선(현재 로그아웃 헤더에 탐색 진입점 부재 문제와 함께).

### L4. 작성자 정체성 / 프로필 공개 (텐션의 핵심 — 결정 선행)
- 커뮤니티 팔로우·크리에이터 신뢰를 위해 **작성자를 공개 표면에 드러낼지**. 0007 익명화와 충돌 → 열린질문 1의 모델 선택에 종속. **결정 없이 착수 금지.**
- 채택 시: `explore_logs`에 표시명/공개 핸들 재도입, 공개 프로필 페이지 `/u/[handle]`, 팔로우 테이블(`follows`). 이건 **별도 ADR + 아마 별도 Phase**.

### L5. 댓글 (보류 후보 — 모더레이션 부담)
- 공개 코스 댓글. 스팸·신고·모더레이션·미성년 안전 부담이 큼. **초기 스코프에서 제외 권장**, 커뮤니티가 자리 잡은 뒤 재검토. (05-F에서도 유보였음.)

## 권장 로드맵 (단계적)

> 원칙: 익명화 텐션(L4)을 건드리지 않고도 갈 수 있는 데까지 먼저 간다 → 커뮤니티감을 저비용으로 확보하고, 정체성 공개는 사용자가 명시 결정한 뒤 별도 Phase.

- **C-1 (참여 최소셋, 익명 유지 가능):** L1 좋아요+북마크 + L3의 "인기/최신 정렬"만. 작성자는 여전히 익명이어도 성립(좋아요는 콘텐츠에 대한 것). → 커뮤니티 첫 심장박동. 가장 빠른 ROI.
- **C-2 (전환 루프):** L2 "이 코스 따라하기" + `forked_from` 인기 신호 + L3 `/community` IA 정식화.
- **C-3 (정체성 — 결정 후에만):** L4 작성자 표시/공개 프로필/팔로우. **열린질문 1 확정 + 신규 ADR 선행.** 큰 작업.
- **C-later:** L5 댓글 — 커뮤니티 규모·모더레이션 체계 갖춘 뒤.

> **STEP 0 선행 유효:** 어느 단계든 **공개 실데이터(공개 코스 8~15건)**가 없으면 커뮤니티는 빈 껍데기다. `03-adsense.md` STEP 0 / PROGRESS `## 다음` 1번이 여전히 공통 선행조건.

## 완료 기준 (C-1 기준)
- 로그인 유저가 공개 코스/장소에 좋아요·북마크 → 재방문 시 유지, `/saved`에서 확인.
- 좋아요 카운트가 공개 표면에 보이되 **누가 눌렀는지는 anon/타인에게 비노출**.
- `/explore`(또는 `/community`)에서 인기/최신 정렬 동작.
- 좋아요/북마크 대상이 `visibility=public` 로그로 제한됨(RLS로 강제) — build-qa/reviewer 확인.
- (C-2) 공개 코스 "따라하기" → 내 커플 초안 생성(장소/순서만, memo/사진 미복제).

## 05와의 관계
- 05-**F**(좋아요·따라하기·댓글·북마크/팔로우)와 05-**Phase 3**(큐레이션·좋아요·따라하기)는 이 문서로 **통합·구체화**된다. 05는 SEO/콘텐츠 볼륨(A·B·C·E)이 주력이었고, 이 문서는 그 위의 **참여·커뮤니티 레이어**다. 05 열린질문 5(좋아요/조회수)·6(댓글)·7(따라하기)의 답이 여기 로드맵.
- 05-C(지역 탐색, 방금 Phase 2-C로 착수/완료 예정)는 L3 커뮤니티 피드의 지역 탭으로 흡수된다.

## 열린 질문 (사용자 결정 필요)
1. **[핵심] 작성자 정체성 모델** — 커뮤니티에서 작성자를 어떻게 다룰지. 0007 완전익명과 충돌하므로 반드시 택1:
   - (a) **완전 익명 유지** — 좋아요/따라하기는 "콘텐츠"에 대한 것으로만. 팔로우·크리에이터 없음. 프라이버시 최상, 커뮤니티감 약함.
   - (b) **선택적 공개 핸들** — 유저가 원하면 커플/개인 표시명(닉네임 아닌 별도 공개 핸들)을 켤 수 있음(opt-in). 기본은 익명. 절충안.
   - (c) **실명(닉네임) 노출** — 0007 뒤집고 explore에 닉네임 복원. 커뮤니티감 최강, 프라이버시 후퇴.
   → 권장: **(b) opt-in 공개 핸들** — 커뮤니티감과 프라이버시 균형. 단 C-1/C-2는 (a)로도 가능하니 이 결정은 C-3 전에만 필요.
2. **이번에 어디까지** — C-1(좋아요·북마크)만 먼저? C-2(따라하기)까지? C-3(정체성)은 결정·별도 Phase가 맞나?
3. **좋아요 신원 저장 범위** — 카운트만 공개는 확정. 그런데 "누가 좋아요/북마크했는지"를 **작성자 본인에게는** 보여줄지(알림성) 아니면 순수 카운트만 둘지.
4. **댓글(L5)** — 초기 제외 권장에 동의? 아니면 커뮤니티에 필수로 볼지.
5. **`/community` 신설 vs `/explore` 확장** — 별도 라우트를 팔지, `/explore`에 탭으로 얹을지(07 헤더 IA와 연동).
