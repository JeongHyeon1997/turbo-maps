# 콘텐츠 확충 — "더 많은 컨텐츠"를 데이터 없이 지금부터

> 사용자 피드백: **"(서비스에) 더 많은 컨텐츠가 있으면 좋겠다."**
> 연관: `03-adsense.md`(심사에 공개 콘텐츠 필요), `04-public-surface.md`·`05-public-enrichment.md`(공개 표면 — 구조 완성, 실데이터 대기), `06-community.md`(참여/커뮤니티 레이어).
> 이 문서의 새로움: 위 계획들은 **전부 STEP 0 실데이터(공개 로그 8~15건)에 묶여 있다.** 이 문서는 **실데이터 없이도 지금 만들 수 있는 콘텐츠(A트랙)**를 처음으로 정의한다.

## 배경 / 왜 지금
- 공개 표면(랜딩 `/`·`/explore`·`/explore/[id]`·`/places`·`/places/[id]`·`/explore/regions`)의 **기술 구조는 완성**됐으나 **공개 로그 0건**이라 전부 빈 화면이다(PROGRESS `## 다음` 1번 = 최대 blocker).
- AdSense 심사의 단일 실질 blocker도 "실제 공개 콘텐츠 부재"(`03` STEP 0)다. thin/empty 상태로 신청하면 반려된다.
- **"더 많은 콘텐츠" 요구를 실데이터 대기와 분리해야 한다.** 실데이터는 사용자 작업(비동기·미정 시점)이지만, **에디토리얼/정적 콘텐츠는 지금 세션부터 코드로 만들 수 있고**, 그 자체가 AdSense 심사 품질(사이트 목적 명확성·내비게이션 깊이·오리지널 콘텐츠)과 SEO(롱테일 랜딩·크롤 URL 수)에 직접 기여한다. 즉 **A트랙은 최대 blocker를 부분적으로 우회하는 유일한 트랙**이다.

## 분류 (A / B / C)

각 후보: 가치(AdSense 심사·SEO·사용자) / 비용 / 데이터 의존.

### B. 실데이터(STEP 0)가 쌓여야 살아나는 콘텐츠 (구조 완성, 데이터 대기 — 중복 금지)

- 해소 조건: `03` STEP 0(공개 코스 8~15건, 사용자 작업) + `0009` 라이브 적용(Blocked). → 이 문서 범위 아님, 그대로 `03`/`05` 따름.

### C. 새 기능성 콘텐츠 (기존 계획으로 흡수 — 중복 금지)
- 참여/커뮤니티(좋아요·북마크·따라하기·정렬·프로필) = **`06-community.md`가 소유**(C-1~C-3). 이 문서에서 재정의하지 않는다.

## 데이터모델 (A트랙 — DB 불필요)
에디토리얼/정적 콘텐츠는 **DB·RLS·마이그레이션 불필요.** repo 내 버전관리되는 콘텐츠로 둔다.
- **선택지 (web-dev 판단):** (a) `apps/web/src/content/guides/*.mdx` + 파일시스템 로드, 또는 (b) `apps/web/src/content/guides.ts` 타입드 데이터 배열(`{ slug, title, excerpt, cover, body, publishedAt, tags }`). FAQ는 `faq.ts` 데이터 배열.
- 권장: 초기엔 (b) 타입드 데이터가 가볍고 검증 쉬움. 아티클 수가 늘면 (a) MDX로 이관.
- 스키마 소유: 순수 web 로컬 콘텐츠라 `@maps/shared` Zod 불필요(공유 도메인 아님). 웹 로컬 타입으로 충분.
